# Usage Signals

Usage signals show whether a tool, template, workflow, product, platform or capability is being used.

They are often the easiest adoption signals to collect.

They are also among the easiest to misinterpret.

## Core principle

Usage is evidence of activity.

It is not proof of adoption quality, value creation, workflow change or responsible use.

```text
Usage can tell us that something is being used.
It cannot tell us whether it is being used well.
```

## What usage signals can show

Usage signals can indicate:

* awareness
* access
* initial interest
* early activation
* repeated interaction
* concentration of use in specific teams
* concentration of use in specific roles
* emerging habits
* possible workflow relevance
* possible dependency on a tool or template

## What usage signals cannot prove

Usage signals do not prove:

* workflow relevance
* task success
* output quality
* user proficiency
* value creation
* risk control
* trust
* appropriate use
* sustainable adoption
* readiness for scaling

## Common usage signals

Examples include:

* active users
* frequency of use
* number of sessions
* number of logins
* number of prompts submitted
* number of AI-supported drafts
* number of completed digital tasks
* number of accessed knowledge assets
* number of workflows using a new template
* number of downloaded or reused templates
* share of target users who tried the capability
* number of recurring users after 30, 60 or 90 days
* usage by role, team, workflow or location
* usage by task type or use case category

## Stronger usage signals

Weak usage signal:

```text
Total number of users.
```

Stronger usage signal:

```text
Share of target users who completed first meaningful use in a relevant workflow.
```

Weak usage signal:

```text
Number of prompts submitted.
```

Stronger usage signal:

```text
Number of AI-supported workflow tasks completed with documented review.
```

Weak usage signal:

```text
Template downloads.
```

Stronger usage signal:

```text
Templates reused in real recurring workflows with acceptable quality after review.
```

## Usage signal maturity

Usage signals become more useful when they are segmented.

### Level 1: Raw usage

```text
How many people used it?
```

### Level 2: Target-group usage

```text
Are the intended roles using it?
```

### Level 3: Workflow usage

```text
Is it being used in the workflows where adoption matters?
```

### Level 4: Repeat usage

```text
Do people come back after first use?
```

### Level 5: Quality-connected usage

```text
Is usage producing acceptable outputs or better workflow results?
```

### Level 6: Decision-connected usage

```text
Does usage help us decide what to scale, pause, redesign or clarify?
```

## Related signal categories

Usage signals should usually be interpreted together with:

* activation signals
* workflow signals
* quality signals
* value signals
* risk signals
* trust signals
* scaling signals

Usage alone is rarely enough.

## Interpretation patterns

### High usage may mean adoption is working

High usage may indicate:

* strong relevance
* low friction
* clear value
* good workflow fit
* useful enablement
* strong manager support
* growing habit formation

### High usage may also hide problems

High usage may also indicate:

* pressure to use the tool
* inefficient repeated attempts
* unclear alternatives
* novelty effect
* poor output quality requiring more attempts
* unmanaged experimentation
* lack of governance
* tool use without workflow change

### Low usage may mean adoption is weak

Low usage may indicate:

* low awareness
* poor access
* weak enablement
* poor workflow fit
* low trust
* unclear permission
* low perceived usefulness
* lack of manager support

### Low usage may also be appropriate

Low usage may also indicate:

* the use case is not relevant
* the workflow is low frequency
* users are applying good judgment
* risk sensitivity is high
* adoption is intentionally limited
* the tool is not needed for that task

## Decision logic examples

### Usage is low

```text
If usage is low,
then awareness, access, relevance, trust or usability may be weak.

Action:
Check whether the target users understand the use case and can apply it in their actual workflow.
```

### Usage is high but quality is low

```text
If usage is high but quality is low,
then adoption volume may be outpacing proficiency.

Action:
Improve guidance, examples, input quality and review criteria before scaling further.
```

### Usage is high in one team but low elsewhere

```text
If usage is high in one team but low elsewhere,
then adoption may depend on a local champion rather than on a reusable operating pattern.

Action:
Identify what the successful team does differently and translate it into reusable enablement.
```

### Usage grows but old workflows remain

```text
If usage grows but old workflows remain unchanged,
then the tool may be added on top of existing work instead of changing the work.

Action:
Redesign the workflow around the capability.
```

### Usage increases but trust declines

```text
If usage increases while trust declines,
then adoption may be driven by pressure or unclear accountability.

Action:
Investigate role clarity, perceived risk and psychological safety.
```

## Countermetrics

Usage signals should be paired with countermetrics.

| Usage metric             | Countermetric             |
| ------------------------ | ------------------------- |
| More active users        | Quality after review      |
| More prompts submitted   | Output usefulness         |
| More AI-supported drafts | Review rejection rate     |
| More template downloads  | Actual workflow reuse     |
| More sessions            | Task completion           |
| More self-service        | Unresolved edge cases     |
| More automation          | Unclear accountability    |
| More recurring users     | Sustained value and trust |

## Questions for adoption reviews

Use these questions when reviewing usage signals:

```text
Who is using it?
Who is not using it?
In which workflows is it used?
Is usage repeated or one-time?
Is usage connected to quality?
Is usage connected to value?
Is usage creating risk?
Is usage replacing old work or adding work?
Is usage concentrated in one team or role?
What would we do if usage increased or decreased?
```

## Common mistakes

Avoid these mistakes:

* treating logins as adoption
* treating usage volume as value
* ignoring role-specific usage
* ignoring workflow context
* celebrating usage before quality is stable
* scaling based on usage alone
* not checking whether old workarounds remain
* measuring available activity instead of meaningful adoption

## Final principle

Usage signals are useful, but they are not enough.

```text
Usage is the beginning of the adoption question, not the answer.
```

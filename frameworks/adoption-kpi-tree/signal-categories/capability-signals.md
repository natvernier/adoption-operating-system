# Capability Signals

Capability signals show whether people understand how to use a new capability appropriately.

They help distinguish confident adoption from superficial usage.

## Core principle

Adoption requires more than access.

People need enough understanding, judgment and confidence to use the capability in the right context, with the right limits and the right review behavior.

```text id="zmfeu4"
Capability is the difference between using something and using it well.
```

## What capability signals can show

Capability signals can indicate:

* understanding
* judgment
* appropriate use
* confidence
* skill development
* role readiness
* maturity of practice
* reduction of basic errors
* readiness for more complex workflows
* whether enablement is working
* whether people understand limits and responsibilities

## What capability signals cannot prove alone

Capability signals do not automatically prove:

* sustained adoption
* value creation
* workflow integration
* output quality
* governance maturity
* readiness for scaling

Capability must be interpreted together with behavior, workflow, quality, risk and trust signals.

## Common capability signals

Examples include:

* users can explain suitable and unsuitable use cases
* users know which review steps are required
* users can explain escalation rules
* managers can explain how the new workflow changes team routines
* reviewers see fewer basic misunderstandings
* questions become more specific over time
* teams create better input briefs
* users choose the right template or workflow pattern
* users can explain where human judgment remains necessary
* users understand the difference between experiment, draft, reviewed output and approved output
* users can identify when not to use the tool
* users can adapt examples to their own context
* teams create reusable prompts, checklists or workflow patterns
* repeated basic support questions decrease over time

## Capability signal maturity

### Level 1: Awareness

```text id="5nrxy8"
People know the capability exists.
```

### Level 2: Basic understanding

```text id="0gxjnp"
People understand what the capability is for.
```

### Level 3: Guided use

```text id="j9vwf4"
People can use it with instructions, examples or support.
```

### Level 4: Independent appropriate use

```text id="jmy5ao"
People can choose when and how to use it in their own workflow.
```

### Level 5: Judgment

```text id="9898b6"
People understand limits, risks, review needs and escalation paths.
```

### Level 6: Transfer

```text id="xou5aa"
People can adapt the capability to adjacent workflows without weakening quality or governance.
```

## Related signal categories

Capability signals should usually be interpreted together with:

* activation signals
* workflow signals
* quality signals
* risk signals
* trust signals
* governance signals
* scaling signals

Capability is especially important when adoption depends on human judgment, not only tool access.

## Interpretation patterns

### Confidence increases but quality does not

This may indicate:

* overconfidence
* weak review understanding
* missing examples
* unclear quality criteria
* insufficient feedback loops
* training focused on tool use instead of judgment

### Questions become more specific

This may indicate:

* capability is increasing
* users are moving from basic awareness to applied judgment
* workflow relevance is becoming clearer
* people are encountering real cases
* enablement can now become more advanced

### Basic questions repeat over time

This may indicate:

* guidance is unclear
* guidance is hard to find
* training is too abstract
* workflow support is missing
* managers are not reinforcing the change
* the system is too complex

### Users avoid complex workflows

This may indicate:

* low confidence
* unclear permission
* missing examples
* fear of risk
* insufficient review support
* lack of manager encouragement
* weak governance clarity

## Decision logic examples

### Confidence without review clarity

```text id="ajx6h9"
If users report confidence but cannot explain review responsibilities,
then adoption may be performative rather than operational.

Action:
Clarify decision rights, quality gates and human judgment points.
```

### Repeated basic questions

```text id="t6dshr"
If users repeatedly ask the same basic questions,
then the guidance is either unclear, hard to find or not embedded in the workflow.

Action:
Improve guidance at the point of work instead of adding another general training.
```

### Specific questions increase

```text id="cr6xuv"
If questions become more specific over time,
then capability may be increasing.

Action:
Use question themes to update examples, advanced guidance and workflow-specific support.
```

### Users avoid suitable high-value use cases

```text id="ripfsa"
If users avoid suitable high-value use cases,
then they may lack confidence, permission or examples.

Action:
Provide role-specific examples, clarify acceptable use and add first-use support.
```

### Reviewers see repeated misunderstandings

```text id="cyvj8v"
If reviewers see repeated basic misunderstandings,
then capability gaps are appearing downstream.

Action:
Update input guidance, examples and training before scaling further.
```

## Countermetrics

Capability metrics should be paired with countermetrics.

| Capability metric              | Countermetric                      |
| ------------------------------ | ---------------------------------- |
| Higher confidence              | Quality after review               |
| More independent use           | Escalation quality                 |
| Fewer basic questions          | Hidden confusion or underreporting |
| More advanced use cases        | Risk signals                       |
| Faster use                     | Error or correction rate           |
| More reusable patterns created | Pattern quality and reuse          |
| Less support needed            | User trust and outcome quality     |

## Questions for adoption reviews

Use these questions when reviewing capability signals:

```text id="qepe7c"
Can users explain suitable and unsuitable use cases?
Can users explain required review steps?
Can users distinguish experiment, draft, reviewed and approved output?
Are questions becoming more specific?
Are basic questions decreasing?
Do users know when to escalate?
Do managers understand their role in adoption?
Do reviewers see fewer basic errors?
Is confidence supported by quality?
Which role needs targeted enablement?
```

## Common mistakes

Avoid these mistakes:

* treating training completion as capability
* treating confidence as proficiency
* ignoring review understanding
* ignoring role-specific capability gaps
* measuring skill without workflow application
* assuming fewer questions always means better understanding
* scaling before capability is stable
* focusing only on tool操作 instead of judgment

## Final principle

Capability signals show whether adoption is becoming competent.

```text id="8x7jux"
Responsible adoption depends on people knowing not only how to use a capability, but when, why and with which limits.
```

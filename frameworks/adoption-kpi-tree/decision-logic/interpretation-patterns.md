# Interpretation Patterns

Adoption signals are rarely self-explanatory.

The same signal can indicate progress, friction, hidden risk or improved visibility depending on context.

Interpretation patterns help teams avoid false conclusions.

They turn adoption measurement from a dashboard exercise into a learning and decision system.

## Core principle

A signal is not an answer.

A signal is a prompt for interpretation.

```text
The same metric can mean different things in different workflows, roles or adoption stages.
```

## Why interpretation patterns matter

Many adoption mistakes happen because teams interpret signals too quickly.

Examples:

```text
High usage is treated as success.
Low usage is treated as resistance.
More escalations are treated as failure.
Fast outputs are treated as productivity.
Training completion is treated as readiness.
Confidence is treated as proficiency.
```

Each of these interpretations may be wrong.

A useful Adoption KPI Tree makes interpretation explicit.

## Pattern 1: High usage does not always mean strong adoption

### Signal

```text
Usage increases.
```

### Possible interpretations

High usage may indicate:

* strong workflow relevance
* curiosity
* pressure to use the tool
* lack of alternatives
* inefficient use
* repeated attempts due to poor output quality
* novelty effect
* unmanaged experimentation

### Better interpretation question

```text
Is usage connected to relevant workflow completion, quality and repeat behavior?
```

### Related signals to check

* task completion
* repeat use
* workflow coverage
* quality after review
* user confidence
* old-workflow persistence
* review effort
* risk indicators

### Decision logic

```text
If usage increases but quality, repeat use or workflow completion does not improve,
then adoption may be active but not yet effective.

Action:
Review workflow fit, output quality and support needs before scaling.
```

## Pattern 2: Low usage does not always mean resistance

### Signal

```text
Usage is low.
```

### Possible interpretations

Low usage may indicate:

* low awareness
* poor access
* unclear permission
* weak workflow relevance
* low trust
* fear of making mistakes
* lack of manager support
* poor timing
* unclear value
* good judgment that the use case is not suitable

### Better interpretation question

```text
Is low usage caused by resistance, friction, irrelevance or appropriate caution?
```

### Related signals to check

* awareness
* access
* first meaningful use
* trust
* manager support
* workflow fit
* support questions
* perceived usefulness
* risk sensitivity

### Decision logic

```text
If usage is low in a high-value workflow,
then investigate whether the issue is relevance, trust, access, permission or capability.

Action:
Do not assume resistance before checking workflow reality.
```

## Pattern 3: More escalations may be progress

### Signal

```text
Escalations increase.
```

### Possible interpretations

More escalations may indicate:

* hidden uncertainty is becoming visible
* people understand when to ask for help
* governance is becoming operational
* the workflow has unclear decision rights
* risk is increasing
* the escalation path is too broad
* users lack confidence
* the policy is too abstract

### Better interpretation question

```text
Are escalations showing failure, learning, risk or improved visibility?
```

### Related signals to check

* escalation type
* escalation severity
* repeated themes
* time to resolution
* ownership clarity
* sensitive use cases
* policy gaps
* user confidence

### Decision logic

```text
If escalations increase after rollout,
then classify escalation types before treating the increase as failure.

Action:
Separate learning signals from risk signals.
```

## Pattern 4: Faster work may shift effort downstream

### Signal

```text
Cycle time decreases.
```

### Possible interpretations

Faster work may indicate:

* genuine efficiency
* better input structure
* improved reuse
* automation benefit
* reduced preparation effort
* lower quality at handover
* more review work downstream
* hidden rework
* premature completion

### Better interpretation question

```text
Did the workflow become faster, or did effort move to another role?
```

### Related signals to check

* review effort
* correction rate
* rework
* handover quality
* output consistency
* reviewer workload
* downstream delays
* quality gate results

### Decision logic

```text
If cycle time improves but review effort increases,
then speed gains may be shifting work downstream.

Action:
Improve pre-review quality gates and input standards.
```

## Pattern 5: Confidence is not the same as proficiency

### Signal

```text

```

### Possible interpretations

* stronger trust
* improved enablement
* more willingness to experiment
* overconfidence
* unclear perception of risk

### Related signals to check

* output quality

* capability signals

### Decision logic

```text
If users report confidence but cannot explain review responsibilities,

```

## Pattern 6: Training completion is not readiness

* compliance
* interest

## Pattern 8: Reuse can create generic outputs

### Signal

```text
Template or pattern reuse increases.
```

### Possible interpretations

Higher reuse may indicate:

* better standardization
* reduced blank-page work
* improved consistency
* faster onboarding
* weak contextual adaptation
* generic outputs
* overstandardization
* loss of audience or use-case fit

### Better interpretation question

```text
Is reuse improving consistency without weakening relevance?
```

### Related signals to check

* use-case fit
* reviewer feedback
* output variation

### Decision logic

```text
If reusable templates are used frequently but outputs become generic,

Add adaptation checkpoints and examples.
```

## Pattern 9: Local success may not be scalable

### Signal

```text
One team shows strong adoption.
```

### Possible interpretations

Local success may indicate:

* strong workflow fit
* effective manager support
* a local champion
* better capability
* stronger trust
* unique team conditions
* hidden support effort
* dependency on one person

### Better interpretation question

```text
Is this success caused by a repeatable pattern or by unique local conditions?
```

### Related signals to check

* manager behavior
* local rituals
* support dependency
* team capability
* workflow similarity
* reusable templates
* adoption beyond champion
* quality stability

### Decision logic

```text
If one team adopts strongly but others do not,
then adoption may depend on local conditions.

Action:
Identify what is repeatable before scaling.
```

## Pattern 10: Policy existence does not mean governance works

### Signal

```text
A policy or guideline exists.
```

### Possible interpretations

Policy existence may indicate:

* formal governance
* documented expectations
* risk awareness
* limited operational clarity
* unclear application
* weak escalation paths
* policy detached from workflow reality

### Better interpretation question

```text
Can people apply the policy in real workflow decisions?
```

### Related signals to check

* repeated governance questions
* escalation path usage
* decision rights clarity
* sensitive use case handling
* exception logs
* review completion
* user understanding
* workflow-specific examples

### Decision logic

```text
If policy exists but teams repeatedly ask how to apply it,
then governance is too abstract.

Action:
Translate policy into workflow-specific examples and decision rules.
```

## Pattern 11: Stable quality may hide low ambition

### Signal

```text
Quality remains stable.
```

### Possible interpretations

Stable quality may indicate:

* responsible adoption
* effective review
* mature practice
* low adoption depth
* low workflow complexity
* limited use cases
* conservative use
* missed value potential

### Better interpretation question

```text
Is quality stable because adoption is mature or because adoption is shallow?
```

### Related signals to check

* workflow coverage
* value indicators
* high-value use case adoption
* user confidence
* repeat behavior

* scaling readiness

### Decision logic


## Pattern 12: No risk signals may mean poor visibility

### Signal

```text
No risk issues are reported.
```

### Possible interpretations

No risk signals may indicate:

* genuinely safe adoption
* low-risk use cases
* strong governance
* underreporting
* unclear escalation paths
* fear of reporting
* lack of monitoring
* hidden workarounds

### Better interpretation question

```text
Are there no risk issues, or are risk issues not visible?
```

### Related signals to check

* escalation awareness
* reporting channels
* psychological safety
* review findings
* informal workarounds
* sensitive use case logs
* governance questions
* audit or review samples

### Decision logic

```text
If no risk signals appear in a complex or sensitive workflow,
then visibility may be weak.

Action:
Check whether escalation paths, review gates and reporting expectations are clear.
```

## How to use interpretation patterns

Use these patterns during:

* adoption reviews
* rollout retrospectives
* governance discussions
* scaling decisions
* KPI tree revisions
* leadership reporting
* pilot evaluations
* workflow redesign sessions

For every important signal, ask:

```text
What else could this mean?
Which related signals should we check?
What would make this interpretation more credible?
What action would be premature?
What action would be justified?
```

## Final principle

Adoption measurement should reduce false certainty.

It should help teams see where work is changing, where signals are ambiguous and where decisions are needed.

```text
A mature adoption system does not only collect signals.
It learns how to interpret them.
```

# README.md

# Decision Logic

Decision logic turns adoption measurement into operating action.

It defines how adoption signals should be interpreted, when they should trigger decisions and what actions should follow.

This folder is part of the Adoption KPI Tree framework.

## Purpose

The purpose of decision logic is to prevent adoption measurement from becoming passive reporting.

A metric is only useful if it helps the organization decide what to do next.

```text
An Adoption KPI Tree is not a reporting structure.
It is a decision structure.
```

## Core idea

Many organizations measure adoption through activity:

* users trained
* tools accessed
* sessions completed
* prompts submitted
* templates downloaded
* pilots launched
* dashboards viewed

These signals can be useful, but they are not enough.

Decision logic asks:

```text
What does this signal mean?
What should we do if it changes?
Who owns the response?
When should we review it again?
What countermetric prevents false success?
```

## Folder contents

This folder contains three supporting files.

### `if-signal-then-action.md`

A practical pattern library for turning adoption signals into actions.

Use this file when you need to define:

```text
If [signal],
then [interpretation],
therefore [action].
```

It includes examples for:

* training without activation
* usage without workflow change
* high usage with low quality
* more escalations after rollout
* confidence without clarity
* speed with downstream effort
* reuse without relevance
* local champion dependency
* policy without operational clarity

### `decision-thresholds.md`

A guide for deciding when a signal becomes actionable.

It defines four threshold levels:

```text
Watch
Investigate
Act
Escalate
```

Use this file when you need to clarify:

* when to keep observing
* when to investigate
* when to change the workflow
* when to escalate to governance or leadership
* when a signal is strong enough to influence scaling

### `interpretation-patterns.md`

A guide for avoiding false conclusions.

It explains why adoption signals are rarely self-explanatory.

Use this file when you need to interpret signals such as:

* high usage
* low usage
* more escalations
* faster cycle time
* higher confidence
* high training completion
* increased questions
* increased reuse
* local adoption success
* no visible risk signals

## How decision logic fits into the Adoption KPI Tree

The Adoption KPI Tree connects:

```text
Strategic ambition
→ Adoption outcomes
→ Behavioral and workflow signals
→ Quality, value and risk indicators
→ Operating actions
```

Decision logic sits between signals and operating actions.

It helps teams avoid jumping directly from a metric to a conclusion.

```text
Signal
→ Interpretation
→ Threshold
→ Action
→ Owner
→ Review rhythm
```

## Decision logic pattern

Use this basic structure:

```text
If [signal],
then [interpretation],
therefore [action].
```

Use the extended structure when ownership and review are needed:

```text
If [observable signal appears],
then this may indicate [interpretation],
therefore we will [action],
owned by [owner],
reviewed in [forum],
by [timeframe].
```

## Example

```text
If usage increases but review rejection rate also increases,
then adoption volume may be outpacing proficiency.

Action:
Pause expansion, improve input quality, clarify review criteria and update examples.
```

This is stronger than reporting:

```text
Usage increased by 34%.
```

Because the signal now supports a decision.

## Decision action types

Decision logic usually leads to one of these action types:

| Action type | Meaning                                                     |
| ----------- | ----------------------------------------------------------- |
| Scale       | Expand what works                                           |
| Pause       | Stop expansion until quality, trust or risk improves        |
| Redesign    | Change the workflow, template, handover or process          |
| Clarify     | Improve decision rights, ownership or criteria              |
| Train       | Build targeted capability                                   |
| Govern      | Add or improve guardrails, review gates or escalation paths |
| Investigate | Gather more evidence before acting                          |
| Standardize | Turn a successful pattern into reusable guidance            |
| Stop        | End what is not useful, safe or worth maintaining           |
| Observe     | Continue monitoring until the signal stabilizes             |

## Good decision logic should define

For every important adoption signal, define:

```text
Signal:
What did we observe?

Interpretation:
What might it mean?

Threshold:
Is this a Watch, Investigate, Act or Escalate signal?

Action:
What should happen next?

Owner:
Who is responsible?

Forum:
Where is it reviewed?

Countermetric:
What prevents one-sided optimization?

Review rhythm:
When do we revisit the signal?
```

## Common anti-patterns

Avoid these patterns.

### KPI without action

```text
Monthly active users increased.
```

Better:

```text
If monthly active users increase but repeat workflow use remains low,
then adoption may be shallow.

Action:
Review workflow fit and first meaningful use.
```

### Threshold without owner

```text
If review issues increase, improve quality.
```

Better:

```text
If the same review issue appears in two review cycles,
then the workflow owner and reviewer lead update the input criteria before the next rollout wave.
```

### Usage without countermetric

```text
If usage increases, scale.
```

Better:

```text
If usage increases and quality, risk and repeat-use signals remain stable,
then consider scaling.
```

### Risk treated only as failure

```text
If escalations increase, the rollout failed.
```

Better:

```text
If escalations increase,
then classify escalation types before deciding whether this is failure, learning or improved visibility.
```

## Review questions

Use these questions in adoption reviews:

```text
What changed in the work system?
What does this signal probably mean?
What else could it mean?
Which related signals should we check?
Is this a Watch, Investigate, Act or Escalate signal?
Who owns the response?
What action should happen next?
What countermetric should we monitor?
When should we review this again?
```

## Relationship to operating rituals

Decision logic should be reviewed in recurring operating rituals.

Possible review rhythms:

```text
Weekly during rollout
Biweekly during early adoption
Monthly during stabilization
Quarterly during scaling or portfolio review
```

The review should not only ask:

```text
Did the numbers improve?
```

It should ask:

```text
What did we learn about the work system?
What should we change because of what we learned?
```

## Final principle

The purpose of adoption measurement is not to prove that transformation activity happened.

The purpose is to help the organization decide what to scale, pause, redesign, clarify, govern or stop.

```text
The value of an adoption KPI is not the number itself.
It is the action the organization is willing to take when the signal changes.
```

# Adoption KPI Tree Workshop Template

Use this template to facilitate a structured workshop for creating an Adoption KPI Tree.

The goal of the workshop is not to create a perfect dashboard.

The goal is to define which adoption signals matter, how they should be interpreted and which operating decisions they should trigger.

## Purpose

This workshop helps a team move from abstract adoption goals to measurable adoption signals and decision logic.

It is designed for AI adoption, digital transformation, product rollouts, platform adoption, internal enablement initiatives or workflow redesign.

The central workshop question is:

```text
What would we need to see in the work system to know that adoption is actually happening?
```

## Recommended duration

Choose one format:

```text
[ ] 45 minutes: lightweight signal mapping
[ ] 60 minutes: standard KPI tree draft
[ ] 90 minutes: full KPI tree with decision logic
[ ] 120 minutes: KPI tree plus ownership, review rhythm and next actions
```

## Recommended participants

Include people who understand strategy, work reality, quality, risk and enablement.

Suggested roles:

* adoption lead
* workflow owner
* product or platform owner
* manager or team lead
* practitioner or end user representative
* reviewer or quality owner
* governance or risk owner
* enablement or training lead
* data or analytics owner
* communication or change lead

Not every workshop needs every role.

The workshop should include enough perspectives to avoid measuring adoption only from a tool, leadership or communication perspective.

## Workshop inputs

Prepare these materials before the workshop:

```text
[ ] Description of the initiative, workflow, product or capability
[ ] Target roles or user groups
[ ] Known adoption goals
[ ] Known risks or governance requirements
[ ] Existing usage or performance data, if available
[ ] Known friction points
[ ] Existing workflow or process description
[ ] Current training, communication or enablement materials
[ ] Existing quality or review criteria
```

## Workshop outputs

By the end of the workshop, the group should have:

```text
[ ] Strategic adoption ambition
[ ] 3–5 adoption outcomes
[ ] Behavioral and workflow signals
[ ] Quality, value and risk indicators
[ ] Countermetrics
[ ] If-signal-then-action decision logic
[ ] Signal owners
[ ] Review rhythm
[ ] Open questions and next actions
```

## Facilitation principle

Do not start with available data.

Start with the work system.

The first question is not:

```text
What can we measure?
```

The first question is:

```text
What would meaningful adoption look like in real work?
```

Available data matters later.

But if the workshop starts with available data, the team may overmeasure what is easy and undermeasure what is important.

# Workshop Agenda

## 1. Opening and framing

Recommended time:

```text
5 minutes
```

### Facilitator script

```text
Today we are not building a dashboard.

We are defining what adoption would need to look like in the work system.

The goal is to connect strategic ambition, workflow behavior, quality, value, risk and operating decisions.

A useful Adoption KPI Tree does not measure everything.

It helps us decide what to scale, pause, redesign, clarify, govern or stop.
```

### Key principle

```text
An Adoption KPI Tree is not a reporting structure.
It is a decision structure.
```

## 4. Define adoption outcomes

Recommended time:

```text
10–15 minutes
```

Adoption outcomes describe what should become true if adoption works.

They are more concrete than the strategic ambition, but still outcome-oriented.

### Prompt

```text
What should become true if adoption works?
```

### Examples

```text
Relevant workflows include the new capability where useful.
Teams know when and how to use it appropriately.
Human judgment points are clear.
Review effort becomes more focused.
Outputs become more consistent.
Risky use cases are escalated earlier.
Successful patterns are reused across teams.
Old workarounds decrease over time.
```

## 5. Map behavioral and workflow signals

Recommended time:

```text
15–20 minutes
```

Behavioral and workflow signals show whether adoption is appearing in everyday work.

### Main prompt

```text
What would we need to see people doing differently?
```

### Supporting prompts

```text
What would first meaningful use look like?
What would repeat use look like?
What would show that the workflow changed?
What would show that old workarounds are decreasing?
What would show that people understand when and how to use the new capability?
What would show that adoption is relevant, not symbolic?
```

### First meaningful use

A login is not first meaningful use.

A training attendance is not first meaningful use.

First meaningful use is the first completed action in a real workflow that reflects the intended behavior change.

## 7. Define countermetrics

Recommended time:

```text
10 minutes
```

Countermetrics prevent one-sided adoption measurement.

For every growth or value metric, ask what could go wrong if this metric improves too quickly or without enough context.

### Prompt

```text
If this metric improves, what should we watch to make sure we are not creating hidden damage?
```

### Examples

| Growth or value metric   | Countermetric                       |
| ------------------------ | ----------------------------------- |
| More AI-supported drafts | Higher review rejection rate        |
| Faster cycle time        | More downstream corrections         |
| More self-service        | More unresolved edge cases          |
| More automation          | More unclear accountability         |
| More reuse               | More generic or low-context outputs |

## 8. Create if-signal-then-action logic

Recommended time:

```text
15–25 minutes
```

This is the most important part of the workshop.

A KPI is only useful if it can change a decision, conversation or action.

### Core pattern

```text
If [signal],
then [interpretation],
therefore [action].
```

### More detailed pattern

```text
If [observable signal appears],
then this may indicate [interpretation],
therefore we will [action],
owned by [owner],
reviewed in [forum],
by [timeframe].
```

### Examples

```text
If training completion is high but first meaningful use is low,
then enablement is not translating into real workflow behavior.

Action:
Replace generic training with role-specific examples, practice tasks and manager prompts.
```

```text
If usage increases but review rejection rate also increases,
then adoption volume may be outpacing proficiency.

Action:
Pause expansion, improve input quality, clarify review criteria and update examples.
```

```text
If escalations increase after rollout,
then hidden uncertainty may be becoming visible.

Action:
Classify escalation types before treating the increase as failure.
```

# Facilitation Notes

## Keep the group away from generic KPIs

If the discussion becomes too generic, ask:

```text
Where would we see this in the actual workflow?
```

## Push beyond activity metrics

If the group proposes only activity metrics, ask:

```text
What would this prove?
What would it not prove?
What decision would this support?
```

## Make risk discussable

If risk is treated only as a blocker, ask:

```text
Could this risk signal also be a learning signal?
What would we need to classify before deciding?
```

## Watch for false success

If usage is treated as success, ask:

```text
What countermetric would tell us whether this success is creating hidden cost?
```

## Translate abstract ambition into behavior

If the ambition is too vague, ask:

```text
What would people do differently next month if this worked?
```

## Connect every signal to action

If a KPI has no action, ask:

```text
What would we do if this number changed?
Who would own that decision?
```
